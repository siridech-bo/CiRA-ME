Install CiRA ME

Machine Intelligence for Edge Computing — a 4-container Docker stack you can install in minutes.

4 steps to a running install
Validate your box  →  Install images  →  Start containers  →  Log in at http://localhost:3030

Step 1 — Validate

Step 1 of 4

Pre-flight check (zero side effects)

Confirms Docker, disk, ports, tarballs, and GPU driver are all in a state install.sh will accept. Nothing is changed — just diagnosed.

cd deployment
bash validate.sh
cd deployment
validate.bat

Expected output — every check should be PASS or WARN:

== User environment ==   PASS  not running as root
== Docker ==             PASS  docker daemon reachable
                         PASS  docker compose v2 plugin present
== GPU ==                PASS  GPU detected: RTX 5070 Ti (driver 576.88)
                         PASS  nvidia container runtime registered
== Disk ==               PASS  1570 GB free
== Ports ==              PASS  port 3030 free (for frontend)
                         PASS  port 5100 free (for backend)
                         ...

Summary: 11 PASS, 1 WARN, 0 FAIL
Ready for install. Next: bash install.sh
Any FAIL? Do this instead
The FAIL messages include the exact fix command. Copy-paste it, re-run validate.sh, and only continue when everything is PASS or WARN.

Step 2 — Install

Step 2 of 4

Load Docker images

Loads all 4 tarballs, verifies each landed as a tagged image, creates data folders with correct ownership. Safe to re-run.

bash install.sh
install.bat

Typical runtime: 3–5 minutes (mostly loading the 14 GB backend image).

Do NOT run with sudo (Linux)
The install script refuses. Root-owned data files break every subsequent command. Add your user to the docker group instead: 
sudo usermod -aG docker $USER
newgrp docker

Step 3 — Start

Step 3 of 4

Start the containers

Pick GPU or CPU based on your hardware. The start script checks ports and (in GPU mode) verifies the NVIDIA runtime is registered before doing anything.

With GPU recommended if driver ≥ 550

bash start.sh
start.bat

CPU-only works on any machine

bash start-no-gpu.sh
start-no-gpu.bat
First boot takes 60–90 seconds
The backend runs SQLite migrations and imports PyTorch on first start. Give it 2 minutes before assuming something is wrong. Watch progress: 
bash logs.sh backend

Your data (data/, datasets/, watcher-data/) lives on the host disk and survives every update. The 4 container images can be replaced any time — data stays.

Backup your data

Run before any risky operation (update, uninstall, rebuild). Writes a single tarball you can copy off the machine or restore later.

# default destination: backups/cirame-YYYYMMDD-HHMMSS.tar.gz
bash backup.sh

# or with an explicit destination
bash backup.sh /path/to/dest.tar.gz
REM default destination: backups\cirame-YYYYMMDD-HHMMSS.zip
backup.bat

REM or with an explicit destination
backup.bat C:\path\to\dest.zip

The script optionally stops containers first so the SQLite DB is quiesced — recommended.

Restore from backup

tar xzf backups/cirame-20260704-201530.tar.gz -C /target/deployment/folder
powershell -Command "Expand-Archive -Path 'backups\cirame-20260704-201530.zip' -DestinationPath 'C:\target\folder'"

Update to a new version

Two paths — both preserve your data and auto-snapshot the database before touching anything.

Option A — In-place (recommended)

Drop the new .tar files into your existing deployment folder, then:

bash update.sh
bash start.sh
update.bat
start.bat

The update script:

  1. Snapshots data/database/ to data/database.backup.<timestamp>/
  2. Stops the running app
  3. Loads and verifies each new image
  4. Prunes old image layers
Rollback if the new version fails to boot

The update script prints the exact rollback commands. Example:

bash stop.sh
rm -rf data/database
mv data/database.backup.20260704-201530 data/database
bash start.sh
stop.bat
rmdir /s /q data\database
move data\database.backup.20260704-201530 data\database
start.bat

Option B — New folder + migrate

If you prefer keeping each release in its own folder:

# in the NEW folder
bash migrate.sh /path/to/old/deployment
bash install.sh
bash start.sh
REM in the NEW folder
migrate.bat "C:\path\to\old\deployment"
install.bat
start.bat

migrate.sh refuses if source and target are the same folder, and aborts on any copy failure — no partial migrations.

Uninstall

By default, uninstall preserves your data on disk. Only containers and images are removed.

# removes containers + images, keeps data
bash uninstall.sh

# ALSO deletes data/, datasets/, watcher-data/ (unrecoverable)
bash uninstall.sh --purge-data
REM removes containers + images, keeps data
uninstall.bat

REM ALSO deletes data\, datasets\, watcher-data\ (unrecoverable)
uninstall.bat --purge-data
Typed-phrase confirmation
Uninstall asks you to type REMOVE for the safe path, or PURGE for the destructive one. "yes/no" prompts get answered on autopilot — a typed phrase forces a pause.

Old NVIDIA driver? (515.x)

The shipped backend is built with CUDA 12.8, which needs NVIDIA driver 550 or newer. If nvidia-smi shows a driver in the 515.x–545.x range, you have three options:

A. Update the driver
Cleanest long-term. Requires a reboot. Talk to whoever else uses the box first.
B. Run in CPU mode
Zero extra work — use start-no-gpu.sh instead. Everything works, but TimesNet training and TI model conversion run slower on CPU.
C. Rebuild the backend for CUDA 11.7
Runs at full GPU speed on the old driver. Rebuild takes ~15 minutes and needs the source (not just the deployment tarballs): 
# from the repo root (not the deployment folder)
docker build \
    --network=host \
    --build-arg TORCH_VERSION=2.0.1 \
    --build-arg TORCH_CUDA=cu117 \
    -t cirame-backend:latest \
    backend/

# if you want to redistribute:
docker save cirame-backend:latest -o deployment/cirame-backend.tar
Every feature (TimesNet training, GPU inference, TI export) works identically. Only the CUDA runtime version differs.

Troubleshooting

Always run validate.sh first — it names the exact fix for every common problem.

Docker not running / images fail to load 
docker info
df -h  # need 30 GB free

On Linux, if you see permission denied: add your user to the docker group.

Port 1883 already in use (Linux)

Common on Ubuntu/Debian — a host mosquitto package is running as a systemd service. Stop it:

sudo systemctl stop mosquitto
sudo systemctl disable mosquitto
GPU not detected / "could not select device driver" 
nvidia-smi                                       # does the host see the GPU?
docker info | grep nvidia                        # is the runtime registered?
docker run --rm --gpus all nvidia/cuda:12.4.0-base-ubuntu20.04 nvidia-smi

If the last command fails: reinstall nvidia-container-toolkit and restart Docker.

Backend crashes with CUDA error on first inference

Your NVIDIA driver is too old (needs 550+). See Old NVIDIA driver? above — either rebuild for cu117 or use start-no-gpu.sh.

Frontend loads but shows connection errors

Backend is still starting up (SQLite migrations + torch import take ~90s on first boot). Wait 2 minutes, then check backend logs:

bash logs.sh backend --no-follow | tail -50
Cannot access from other machines on the LAN
  1. Firewall must allow ports 3030, 1883, 9001.
  2. Use the server IP, not localhost: http://192.168.x.x:3030
Data missing after upgrade

This shouldn't happen — data is on the host disk in ./data/ and ./datasets/. Common causes:

  1. You extracted the new release to a different folder → run migrate.sh <old_folder>.
  2. You deleted data/ or datasets/ by mistake → restore from your most recent backup.sh output.
