## Part 6 — Iterations after the initial write-up

### `e9ebec4` — Mixed-type sensor detector

`_check_mixed_type_sensor()` in [data_loader.py](backend/app/services/data_loader.py). Scans every non-time / non-label object-dtype column, tries `pd.to_numeric(errors='coerce')`, and raises `NON_NUMERIC_SENSOR` when the column is **≥80% coercible** with at least one failure. Message names the exact column, offending row number (1-based), and bad value — plus the numeric percentage.

Called from both `load_csv` and `load_text` before the empty-sensor fallback so it catches corrupted columns even when other clean numeric sensors exist. Threshold of 80% prevents false positives on legitimate string columns (part IDs like `A-42`, etc.).

**Verified against** `bad_non_numeric_sensor.txt` — fires `NON_NUMERIC_SENSOR` with `"Column vibration has a non-numeric value on row 4: ERROR"`.

### `edaf100` — Text Import wizard preamble auto-skip

`/api/data/text-sniff` gained a `suggested_skip_rows` field. Counts fields per line using the detected delimiter, finds the most common count > 1 (the "expected" column count), and counts leading lines whose count differs as preamble. Safety cap prevents skipping every line.

Frontend Text Import wizard pre-populates the **Skip N rows from top** input from the sniff response so files with leading `#` comment / title lines Just Work when the user clicks Import with defaults.

**Verified** on `good_with_preamble.log` — auto-suggested `skip_rows = 3`.

### `6989a6d` — Datetime auto-conversion

`_coerce_timestamp_to_seconds()` in `data_loader.py`. When the detected timestamp column holds non-numeric values, parses with pandas' `format='mixed'` engine and rewrites in-place to **seconds since the first sample** (float). Downstream sample-rate detection, feature extraction, and chart scaling all work correctly on datetime-string files.

- Called after timestamp detection in `load_csv`, `load_text`, `load_csv_multiple`.
- Numeric columns pass through unchanged.
- Unparseable values raise `NON_NUMERIC_TIMESTAMP` naming the offending row + value.
- `metadata.time_epoch_start` records the ISO string of the anchor.

**Supported formats** (all verified):
- `2025-04-01 08:00:00`
- `2025-04-01T08:00:01`
- `2025-04-01T08:00:00Z` (UTC)
- `2025-04-01T15:00:00.100+07:00` (with offset)
- `04/01/2025 08:00:00` (US format)

Format Guide dialog updated to mention datetime auto-conversion.

### `d0fc4e2` — Row-click CSV = single-select

Previously, clicking anywhere on a CSV row (icon, filename, empty space) toggled that file in a shared multi-select list — users testing files one at a time would inadvertently build up a selection and see `COLUMN_MISMATCH`. New behavior:

### `1802476` — URL preview response guards

Prevents a hard crash when `/api/data/load-from-url` response body has an unexpected shape. Two changes:

1. Template preview card `v-if="dataPreview"` widened to `v-if="dataPreview?.metadata"` so a partially-shaped response can't tip the renderer into "Cannot read properties of undefined" errors.
2. `fetchFromUrl` validates `response.data` has both `metadata` and `preview` before assigning; on failure surfaces a friendly error in the URL loader panel.

### `db6919b` — Scrub NaN / ±Inf from preview records

**Root cause of a real bug:** Flask's default `jsonify` writes literal `NaN` / `Infinity` tokens for pandas non-finite floats. Standard JSON forbids these tokens, so browsers fail to parse the response body → axios returns undefined → preview render breaks.

`_safe_records()` and `_safe_preview_records()` helpers in `data_loader.py` replace ±Inf with NaN, then object-cast + `where(notna, None)` → Python `None` → JSON `null`.

All **12 preview-emitting call sites** in the loader now route through the helper:
- `load_csv`, `load_csv_multiple`, `load_text`
- `load_json`, `load_cbor`
- folder / partition loaders
- session refresh, load-more, previews

**Reproduced with** CMAPSS FD001 — trailing spaces produced NaN columns that broke the response body. Fixed.

### `7dc4d34` — Drop-all-NaN columns + column_names support

Two related fixes for headerless / whitespace-delimited files:

**Fix A** — `load_text` applies `df.dropna(axis=1, how='all')` post-read to remove fully-empty trailing columns (from trailing whitespace on rows).

**Fix B** — `load_text` accepts an optional `column_names` list. When the count matches the actual column count (after all-NaN cleanup), the auto-generated `col_1..col_N` names get replaced with caller-supplied friendly names. Threaded through `/api/data/load-from-url` and the frontend URL loader.

**CMAPSS canonical schema** added as a constant (`unit_number, time_cycles, setting_1..3, sensor_1..sensor_21`) — 26 names applied to all 5 CMAPSS catalog entries. Now shows meaningful column names, `time_cycles` auto-detected as the X-axis, no trailing NaN artefacts.

### `c746974` — Log Watcher: parse-mode UX overhaul (A + B + C)

**Files:** `backend/app/models.py`, `backend/app/routes/folder_watchers.py`, `backend/app/services/folder_watcher_service.py`, `frontend/src/views/FolderWatcherEditView.vue`

Regex was too heavy for factory operators. Three additions:

**A) NEW `key_value` parse mode** — operator lists column names comma-separated; watcher extracts `key=value` / `key:value` pairs from anywhere in each line. Ignores timestamps, log levels, hostnames, unit suffixes (`45.32°C`, `0.87g`). **Now the default parse mode for new watchers.** Existing watchers keep their configured mode.

**B) Live sample preview panel** — expandable "Try a sample from your log file" panel with a textarea + Test parse button. Backed by new `POST /api/folder-watchers/preview-parse` (4 KB cap, 100-row output cap, NaN → null). Shows extracted columns + up to 5 parsed rows in a table, warning chips for skipped lines, friendly error alert.

**C) Regex templates dropdown** — 5 pre-filled patterns for regex mode (apache, syslog, csv_line, space_sep, factory). Selecting a template auto-fills the regex textarea; editing manually flips template back to Custom.

**DB:** `folder_watchers.parse_columns TEXT` column added with auto-migration.

### `894b764` — Log Watcher edit: clearer sample panel + regex nudge

Follow-up polish: preview panel titled "Try a sample from your log file" (was "Try a sample line") with a bright info alert reminding operators to paste **actual log lines**, not their regex. Textarea now has real log-line placeholder pre-filled. Regex mode header shows an info alert pointing users at Key=Value as the friendlier option. Removed the useless "Key=Value (auto-fill)" regex template that produced a stub regex.

### `b2d6080` — Log Watcher edit: hide Regex behind Advanced toggle

Parse-mode dropdown by default shows only **Key = Value pairs / JSON / CSV**. Small "Show advanced parse modes" text button reveals Regex (labeled "— advanced"). Existing watchers on regex mode auto-reveal the advanced section on load. Hiding advanced while regex is selected snaps back to `key_value` and clears `parse_regex` to prevent an orphaned value in the dropdown.

Factory operators onboarding will never see regex unless they specifically look for it.

### `35b72a3` — Log Watcher edit: auto-detect columns from a sample

New **⚡ Auto-detect columns** button next to Test parse. New `POST /api/folder-watchers/detect-columns` endpoint scans the sample for `word=digit` / `word:digit` patterns (word must start with letter/underscore, up to 64 chars), returns unique keys in first-seen order. Frontend fills the Column names field. Toast reminds operator to trim non-sensor keys (`pid`, `port`, etc.).

Button only appears when parse mode is `key_value` AND Column names is empty AND sample textarea has content. Won't clobber existing user input.

### `b02e147` — Log Watcher edit: TEST PARSE auto-detects when needed

Even better UX: TEST PARSE now silently auto-detects columns if `parse_mode=key_value` AND columns are empty AND sample text has content. The manual Auto-detect button stays for the case of detecting without parsing, but 99% of the time the operator just clicks TEST PARSE and it works.
