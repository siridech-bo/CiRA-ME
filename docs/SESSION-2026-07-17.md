## Bug fixes shipped (workshop debrief)

### Bug 1 — Folder upload appeared to flatten filenames
**Root cause**: purely a display bug in the upload preview list. Files ON DISK were correctly nested; the preview showed `entry.relative_path` (full path) as the row title. When 118 files shared a long common prefix, the container truncated them and every row looked identical → looked like a flattening bug.

**Fix**:
- `DataSourceView.vue:987` — show basename as row title, parent path as smaller secondary text.
- Backend `data_sources.py:185, 319` — kept the defensive `os.path.basename` before `secure_filename` because Firefox / Linux-Chrome edge cases can still leak the full path into `file.filename`.

### Bug 2 — MQTT time-column parsing only worked with "time"
**Root cause**: subscriber's positional-fallback pool included any numeric field, so metadata fields like `_timestamp` or `epoch` took the first fallback slot and offset every real sensor channel by one.

**Fix**:
- `mqtt_publisher.py:265` — expanded `non_sensor` tuple from ~10 aliases to ~25 (adds `ts, datetime, date, epoch, unix_time, elapsed, sec, seconds, ms, milliseconds, t, idx, i, row, row_id`).
- `PublishedAppView.vue:3345` — added `META_FIELD_NAMES` blacklist filtered from `unusedKeys` before positional fallback. Handles dotted keys (`meta._timestamp`) via leaf-segment match.

### Bug 3 — Test Publisher silently stopped
**Root cause**: any WebSocket close/error → `stopMqttPublish()`. No user feedback; operators only noticed when downstream inference went idle.

**Fix**:
- `MqttTestPublisher.vue` — refactored connect+publish loop into `_connectAndPublish()`; on any non-intentional close/error we surface a persistent v-dialog:
  > *"Publishing stopped: <reason>. Sent X/Y rows before the drop. Reconnect will resume from row Z."*
- **Reconnect** reuses cached CSV rows + ws-info (no re-fetch), resumes from `mqttRowIndex`.
- **Stop** resets cursor.

## Features shipped

### Data Source File Manager
One "Manage Files" button next to the existing Browse Files toolbar opens a full-screen modal.

- **Frontend**: [`FileManagerDialog.vue`](../frontend/src/components/FileManagerDialog.vue) (1070 lines), [`FileTreeNode.vue`](../frontend/src/components/FileTreeNode.vue) (158 lines).
- **Layout**: left folder tree, right sortable table with checkbox multi-select (Ctrl/Shift range-select), breadcrumb.
- **Toolbar**: New Folder / Upload / Rename / Move / Copy / Delete / Download. Move/Copy open a folder-picker sub-dialog.
- **Safety**: >10 items require typing DELETE. Inline rename with Enter=save / Esc=cancel.
- **Backend** — 4 new endpoints, all `@login_required` + `validate_path`:
  - `POST /api/data/mkdir`
  - `POST /api/data/rename`
  - `POST /api/data/move`
  - `POST /api/data/copy`
  - `/browse` response gained a `modified` field (Unix seconds).

Existing Browse Files panel unchanged — still the read-optimized "pick a dataset" flow.

### Folder Watcher File Manager (Tier 1 + Tier 2) — the big one

**6 tabs on the watcher edit page** ([`FolderWatcherEditView.vue`](../frontend/src/views/FolderWatcherEditView.vue)):
1. **Config** — original form untouched
2. **Input Files** — drag-drop upload zone + table + preview/delete + `won't be processed` chip for files that don't match the file glob
3. **Output Files** — table + "Download all as .zip" + preview/download/delete
4. **Errors** — table + Retry button; shows the actual exception reason inline
5. **History** — inferred input→output timeline
6. **Advanced (Tier 2)** — container path + host path (from `WATCHER_HOST_BASE_PATH` env) + API upload URL, with copy-to-clipboard

**Backend changes** ([`folder_watchers.py`](../backend/app/routes/folder_watchers.py) + [`folder_watcher_service.py`](../backend/app/services/folder_watcher_service.py)):
- **Input file lifecycle changed**: successful input files now move to `<input>/_processed/<name>` instead of being deleted. Preserves the customer's mental model of "my file is still there".
- **Failure sidecar**: when a file fails processing, a sidecar `<name>.error` is written next to it in `_error/` containing the exception summary + full traceback. Listed inline in the Errors tab; full traceback available via Preview.
- **7 new endpoints** (all `@login_required` + owner-gated):
  - `POST /upload` — multipart file upload
  - `GET /files/<kind>/<name>/download` — binary download (supports `input`/`output`/`error`/`processed`)
  - `DELETE /files/<kind>/<name>` — per-file delete (auto-cleans the sidecar when deleting an error file)
  - `GET /files/output/zip` — bulk zip download
  - `POST /files/error/<name>/retry` — move error file back to input (cleans sidecar)
  - `GET /paths` — Tier 2: container + host + API URLs
  - `GET /history` — inferred input→output mapping (3 statuses: `success`, `archived_no_output`, `output_only`)
- `list_files` for the `input` folder now returns ALL files with a `matches_glob` per-entry flag instead of hiding non-matching uploads. Fixes the "I uploaded a file but it's not showing" pain.
- `list_files` for the `error` folder enriches each entry with `error_reason` (first line of the sidecar) for inline display.
