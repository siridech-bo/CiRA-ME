## 3. Feature contract mismatch — deferred to MOMENT implementation

**Symptom**: dropping a raw CSV into a watcher whose bound model expects 25 features (tsfresh + DSP) fails with:
```
ValueError: X has 4 features, but StandardScaler is expecting 25 features as input.
```

**Root cause**: the watcher processes each CSV row directly (~4 columns), sending it straight to the model — no windowing / feature extraction. Trained models coming out of the standard pipeline expect the full 25-feature vector.

**Where it belongs**: rolled into the MOMENT foundation-model rollout ([PLAN_2026-07-XX_moment-foundation-model.md](PLAN_2026-07-XX_moment-foundation-model.md)). The whole point of foundation models is bypassing feature engineering — a MOMENT-trained model will accept raw windows directly, matching what the watcher already sends. Two birds, one stone.

Until then: users must bind watchers to raw-mode or DL (TimesNet) models, not standard ML pipelines.

## 6. Fast Mode's 5-feature backend parity gap

From Phase 2 QA — 5 tsfresh names are in `Config().TSFRESH_FEATURES` but silently skipped by the backend runtime extractor:
- `length`
- `ratio_beyond_r_sigma`
- `longest_strike_above_mean`
- `longest_strike_below_mean`
- `sum_of_reoccurring_data_points`

Fast Mode computes them; server does not. Safe for closed-loop Fast-train + Fast-inference (Phase 3). Would matter for mixed Fast-train + server-inference — the model would receive 5 zero features on the server side.

**Fix**: either implement them in [`backend/app/services/feature_extractor.py`](../backend/app/services/feature_extractor.py) or remove from `Config().TSFRESH_FEATURES`. ~1 hr.
